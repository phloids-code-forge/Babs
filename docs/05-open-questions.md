## Section 17: Open Questions

The following questions are deliberately unresolved. Some will be answered during architecture design. Others require testing on the target hardware. The architect should propose answers where possible and flag the rest for phloid's decision.

1. **Babs Eval Suite.** Should a version-controlled test suite be a required output deliverable? If yes, define scope: personality compliance, trust tier enforcement, memory retrieval accuracy, task decomposition correctness, Code Before Memory routing. This would serve as the regression test for every model swap, config change, and system update.

2. ~~**Error taxonomy.**~~ **RESOLVED.** See Error Taxonomy specification in Section 9 (01-hardware-and-infrastructure.md). Five categories with subtypes, dead-letter entry schema, and queue lifecycle defined.

3. ~~**Conflicting memories.**~~ **RESOLVED.** See Memory Conflict Resolution in Philosophy Document Section 4 (v1.4). Three-layer system: scoped preferences, newest-wins within scope, ambiguous conflict batching for dashboard review.

4. ~~**Supervisor self-validation.**~~ **RESOLVED.** See Adversarial Validation Protocol in Section 9 (01-hardware-and-infrastructure.md). Applies to Class C decompositions and Tier 2/3 action plans. Not applied to Class A/B (latency cost too high relative to blast radius).

5. **External API outage handling.** When third-party APIs (Alpaca, Gmail, Anthropic) impose their own rate limits or go down, define the handling strategy distinct from Babs' internal rate limiting. Retry? Queue and wait? Notify phloid? Integrate with the circuit breaker protocol in Section 9.

6. **Associative graph edge pruning.** The memory consolidation pipeline deprecates memories but not the links between them. Over months, the SQLite edge table will grow dense with stale connections. Define a pruning strategy for edges, not just nodes.

7. **Drift detection signals.** Measuring "humor frequency" or "agreement rate" requires classifying every response. Define what signals are actually captured, how they are measured, and the compute cost of the classification. If drift detection requires running a model over every response, that has Memory Ledger implications.

8. **Conversation thread metadata.** Each thread should have: a human-readable name (auto-generated or phloid-assigned), creation timestamp, last-active timestamp, thread type (work or buddy), optional project tag, and a periodically updated summary line. Define the schema.

9. **External API health monitoring.** Should the Event Listener proactively monitor external API health (Alpaca responding? Gmail authenticated? Anthropic reachable? OpenRouter reachable? Google AI API reachable?) so Babs can report status before a task fails? For cloud model providers specifically, monitoring should include rate-limit status and remaining budget headroom so the routing layer can preemptively avoid providers that are near their limits.

10. ~~**SGLang vs. vLLM final selection.**~~ **RESOLVED.** vLLM with community NVFP4 patches (Avarok dgx-vllm image or equivalent) is the day-one engine. SGLang is planned as a future addition, particularly for the Supervisor (RadixAttention prefix caching, constrained decoding for structured output). The Nemotron 3 Super HuggingFace model card includes SGLang serving instructions (sglang v0.5.9), confirming compatibility. Evaluate SGLang for the Supervisor role at deployment time, potentially running both engines (SGLang for Supervisor, vLLM for Workers). See Section 1 for full rationale.

11. **G14 auto-boot verification.** Confirm that the ASUS ROG Zephyrus G14 Gen 2 BIOS supports "Restore on AC Power Loss" for automatic reboot after a power outage. If not, Wake-on-LAN from the Spark (requires wired ethernet) is the fallback. This is a prework deliverable (Section 15).

12. **Docker on Ubuntu verification.** Confirm all G14 auxiliary service containers run cleanly under Docker Engine on Ubuntu 24.04 LTS. Document any container images that require modifications for the host environment. This is a prework deliverable (Section 15).

13. **Off-site backup growth projection.** Estimate the 12-month storage growth rate for the critical data set (Semantic Memory, Procedural Memory, audit logs, configuration, consolidated Episodic Memory) against the 2TB Google Drive ceiling. If growth exceeds capacity, recommend a supplemental provider.

14. **Context usage bar (dashboard).** The chat interface should display a persistent progress bar showing real-time token consumption against the Supervisor's usable context ceiling (post-system-prompt, post-memory-retrieval). Color zones: green (fresh), yellow (50%+, extraction should be activating), red (75-80%+, context rot danger zone). The yellow threshold should match the extraction trigger defined in Section 3 so the bar reflects the same signal Babs is acting on. Data source: serving engine response metadata (ground-truth token counts, not estimates). Spec finalized during dashboard design.

15. **PX13 system administration.** Babs should be able to assist phloid with system administration tasks on the PX13 workstation (software updates, configuration management, troubleshooting, etc.) via the Tailscale mesh. Define the access mechanism (SSH, remote execution agent, or similar), permission boundaries, and Trust Tier assignments for sysadmin actions. Spec finalized post-Spark once the core infrastructure is operational.

16. **Buddy-mode retrieval bias tuning.** Define the exact mechanism for partition-aware retrieval. Options: hard filter (personal memories are invisible in work queries), soft bias (personal memories are scored lower), or hybrid (hard filter by default, soft bias when the Supervisor explicitly requests cross-partition context). The choice affects how much Babs' personal knowledge of phloid leaks into work responses, which may be desirable in some cases.

17. ~~**Qwen3.5 NVFP4 kernel stability on GB10.**~~ **PARTIALLY RESOLVED -- UPDATED 2026-03-17.** The Supervisor model was changed from Qwen3.5-122B-A10B to NVIDIA Nemotron 3 Super 120B-A12B NVFP4. Nemotron 3 Super NVFP4 exhibited SM121 instability (`cudaErrorIllegalInstruction`) and fell back to ~1 tok/s via Marlin software emulation. Status as of 2026-03-17: community developer namake-taro published vLLM 0.17.0 MXFP4 patches (github.com/namake-taro/vllm-custom) that fix a shared memory race condition in the Marlin MoE 256-thread kernel on SM121. With these patches and env vars (`VLLM_NVFP4_GEMM_BACKEND=marlin`, `VLLM_TEST_FORCE_FP8_MARLIN=1`, `VLLM_MARLIN_USE_ATOMIC_ADD=1`, `--load-format fastsafetensors`), single-node Spark reaches 14-16 tok/s on Super. Two-node TP=2 reaches 80 tok/s on gpt-oss-120b. 14-16 tok/s is below interactive-use threshold. The `vllm-super` image (avarok v23 base + 0.17.0rc0 wheel) is the current best base; it needs the namake-taro patches applied to get those speeds. Avarok v24 (with proper SM121 FP4 kernel support) remains the unlock for full performance. Nano (65+ tok/s) is the active local model. NemoClaw's nvidia-nim provider covers NVIDIA cloud Super in the interim.

18. **ComfyUI memory budget and model swap lifecycle.** ComfyUI image generation shares the dynamic memory pool with Workers. When the Supervisor dispatches an image generation task: (a) any active Worker model unloads, (b) the diffusion model loads (FLUX-class at ~15-20GB in FP8/NVFP4, plus VAE + CLIP at ~5-8GB), (c) ComfyUI runs the workflow, (d) the diffusion model unloads, (e) memory is reclaimed. Define: the expected swap latency (model unload + diffusion model load + warm-up), memory reclamation guarantees (is all memory freed, or is there fragmentation?), the mechanism (vLLM model unload API, container stop/start, or a separate serving instance for diffusion), and whether the Supervisor remains responsive during the swap (it should, since it stays resident). The daily comic strip workflow (Babs/Phloid character scenes summarizing work progress, dispatched to Discord) is the primary ComfyUI use case.

19. **INT4 (W4A16 Marlin) vs NVFP4 for Workers.** Community benchmarks on the Spark show INT4 via the Marlin FP16 dequantization path significantly outperforming NVFP4 on some workloads (94.6 tok/s vs 68.1 tok/s for Qwen3-Coder-30B-A3B with EAGLE3 speculative decoding). For small Worker models where the quality difference between INT4 and NVFP4 is minimal, INT4 may be the faster option. Benchmark both quantizations at deployment time for each Worker model candidate (Qwen3.5-9B, Qwen3.5-4B, Qwen3.5-35B-A3B) and select based on the quality-to-throughput ratio for Worker tasks specifically (not Supervisor tasks, where quality takes priority).

20. **Tool bridge pattern for sandboxed code.** Workers execute in Docker isolation but may need to access external data sources (APIs, databases, file mounts) during code execution. Define a structured tool bridge pattern that allows sandboxed Jupyter kernels to request external data through a controlled interface rather than having direct network access. This prevents Workers from making unaudited external calls while still enabling data-dependent code execution. Specify the bridge API, allowed data source types, and Trust Tier implications.

21. **Samsung Watch integration.** phloid wears a Samsung Watch. Define a lightweight voice note capture workflow: phloid speaks a quick note into the watch, it is transcribed (on-watch or via Whisper on the G14), and ingested as an Episodic Memory entry tagged with timestamp and location context. Specify the transport mechanism (companion app to phone, then relayed to Babs via Tailscale, or direct if the watch supports it), the minimum viable implementation, and whether this requires a custom app or can leverage existing Samsung/Wear OS APIs.

22. **Driving-mode voice interface.** When phloid is driving, interaction must be fully hands-free and eyes-free. Define a voice-only interaction mode: always-listening wake word or push-to-talk via steering wheel button, STT via Whisper, TTS response via car speakers (Bluetooth audio), no visual dashboard dependency. Specify latency requirements for conversational feel, how this mode interacts with Trust Tiers (voice approval for Tier 2/3?), and the client device (phone as the audio bridge). This is a future expansion item but should have clean integration points in the architecture.

23. **Model scouting eval suite design.** The Model Scouting Pipeline (Section 12) requires a local evaluation suite that Babs runs against candidate models via API. Define: the minimum task count for statistical significance, the specific metrics collected (beyond Procedural Memory compliance and structured output reliability), how to handle models with different tokenizers or chat templates, whether the eval runs against the full model or a quantized version, and how results are stored for historical comparison. The eval suite itself is a Procedural Memory entry, meaning Babs can propose revisions as she learns which metrics best predict Worker quality in practice.

24. **Cloud escalation learning signal.** The Reflection Loop can learn which task types should auto-escalate to cloud models based on correction patterns. Define: what counts as a "correction" for this purpose (phloid explicitly says the answer is wrong? phloid re-asks the same question to a cloud model? phloid edits the output significantly?), the observation window before a pattern is proposed, and how to prevent false positives (a task type that Babs got wrong once should not auto-escalate forever).

25. **Speculative decoding as Class A latency contingency.** Nemotron 3 Super includes built-in Multi-Token Prediction (MTP) with a shared-weight design across prediction heads, enabling native speculative decoding without a separate draft model. MTP has NOT been tested on the Spark yet and should only be evaluated after the base model stability issue (see item 17) is resolved. The 14.8 tok/s baseline was measured WITHOUT MTP enabled. If Class A latency targets (sub-100ms cached, under 1s warm cache, under 2s cold) are still not met with MTP enabled, Eagle3 family or equivalent external speculative decoding remains a secondary contingency. Define: the MTP-enabled throughput gain vs. MTP-disabled, whether MTP is always-on or toggled per request class, and the decision criteria for escalating to external speculative decoding. Document MTP as the primary latency strategy and Eagle3 as the fallback slot.

26. **NemoClaw GPU passthrough to sandbox.** The `openshell gateway start --gpu` flag enables GPU passthrough into the OpenShell k3s environment. This is listed as untested on Spark in the NemoClaw spark-install.md. Current architecture routes all inference through the vllm-local provider (external to the sandbox, hitting vllm-babs Docker container via host.openshell.internal:8000). If GPU passthrough is needed for future in-sandbox inference (e.g., running a small tool-use model inside the sandbox), this flag must be evaluated. Test after the Phase 9 supervisor integration is stable.

27. **OpenClaw agent loop integration pattern.** The Babs supervisor (Python, NATS-based) was built before OpenClaw adoption. Two integration options: (a) Babs supervisor becomes an OpenClaw agent -- supervisor logic is reimplemented as OpenClaw tools and the agent loop replaces the supervisor's run loop. Architecturally cleaner but requires rewriting the supervisor. (b) Babs supervisor orchestrates OpenClaw as a subprocess -- supervisor remains the primary loop, spawns OpenClaw agent calls via SSH (openshell-nemoclaw) when needed. Preserves existing NATS/memory/tool infrastructure with minimal rewrite. Option (b) is the lower-risk starting point. Define the interface contract and evaluate whether option (a) is worth the migration cost after option (b) is working.

28. **OpenShell policy definition for Trust Tiers.** OpenShell policy files define what the sandbox can do (filesystem paths, network egress destinations, process spawning). The mapping from Babs Trust Tier assignments to OpenShell policy rules needs to be defined. Draft: Tier 0 (full autonomy) operations that the agent routinely performs should have standing OpenShell policy rules. Tier 2/3 operations that involve new network destinations or filesystem writes outside established paths should require an OpenShell policy update (which is a version-controlled change requiring phloid review) before they can execute. This makes policy changes explicit and auditable. See docs/03-behavior-and-agency.md for the Trust Tier / OpenShell conceptual split.
